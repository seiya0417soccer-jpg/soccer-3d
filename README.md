90秒スコアアタック × NavMesh連動 落下パズルバトル



―ゲーム概要―

右画面のオリジナル落下パズルでブロックを消すと左画面の勇者が加速し敵をなぎ倒します。そのキル数を競う90秒スコアアタックゲームを製作しました。



―使用技術―

\- Unity / C#

\- NavMeshAgent（敵追跡AI）

\- ScriptableObject（データ駆動設計）



―制作期間―

2025年2月26日〜



―主な実装内容―

\- オリジナル落下パズル（CrossBomb・EKeyBomb搭載）

\- CrossBomb：通常ピースの一部がランダムでこれになり十字方向に連鎖消去

\- EKeyBomb：Eキーで投下、5×5範囲を爆発（パズル救済・勇者5秒停止のトレードオフ）

\- 浮き島検出（FloodFill）→ パラパラ落下アニメーション

\- ブロック消去数に応じた勇者加速システム

\- フィールドサイズはScriptableObjectで管理

―設計方針―

・今回の制作では仕様変更やチーム開発を想定し\*\*疎結合・カプセル化・データ駆動\*\*なアーキテクチャを徹底しました。



\#1. 仲介者（Mediator）パターンによる疎結合



・初期実装では各クラスを直接参照させていましたが、依存関係が複雑化し設計が破綻しました。

・直接やり取りをすることに限界を感じたので試行錯誤していた12時間分を破棄し\*\*メイン管理クラス\*\*を仲介役として導入しました。



―データフロー:―

```

パズル画面（発生源）→ メイン管理（計算ロジック）→ バトル画面（表示）

```



疎結合と一貫性を徹底し、「パズル→管理→バトル」という単方向の流れを守りました。EKeyBombによるデバフも一貫性を守りこのルートで伝えるようにしました。



\#2. 「鉄の掟」によるカプセル化と保護



AIには以下の設計ルールのもと開発させましました。



・「直で触らせない」:\*\* ステージサイズ等の根幹ルールはプロパティ（private set）で保護し、勝手な書き換えを防ぎました。

・「勝手に名乗らせない」:\*\* 本来名乗ってはいけないスクリプトが勝手に名乗っていた（NavMeshAgentをメイン管理も名乗りだした）のでそれを禁じました。



\# 3. ScriptableObjectを活用したデータ駆動設計



カメラ調整などのプランナーが調整したいであろう数値をSOでパブリックに開放し、ゲーム実行中にインスペクターからリアルタイムで微調整できるようにしました。

ステージサイズなど根幹の値は\[SerializeField] + private setで保護し、インスペクターからは調整できるがコードからは書き換えられない設計を採用しました。



―制作してみて―

この経験を通じ、ソフトウェア設計における保守性（疎結合、プレハブ化）の重要性を実体験として深く理解しました。

また、AIへの適切な指示の出し方、使い方、コードを書かせるうえでのリスクを深く理解しました。

ただカレーを作ってと言うのではなく材料と調理器具を与えて初めて動くのだと深く理解しました。

\## AI活用について



\- 設計・依存関係・命名規則・パラメータ管理は自分で行い、コードの実装にAIを活用

\- AIが生成したコードは内容を理解した上で採用・修正

\- AIが勝手に設計を変更しないよう、依存図・命名規則をルールとして明文化して指示

\- ScriptableObjectでデータを分離し、AIがコードから直接数値を触れない設計を採用
